01 PWA · Mobile · FLL Innovation Project 2025–26

Archaeo
Smart

The Problem

Archaeological field teams record findings on paper — fragile, unstructured, impossible to search or share. Our FLL innovation project was to build a solution that actually works in the field, offline, on a phone.

What I Built

A mobile-first Progressive Web App with voice-guided artifact recording, GPS geo-tagging, QR label generation and scanning, canvas drawing for site sketches, CSV data export, map view, and a statistics dashboard — entirely client-side, no server, works offline.

How The System Works
Inputvoice answers, manual form data, GPS location, QR scans, canvas sketches
Storageartifact records are serialized locally so the app still works without internet
RetrievalQR codes and searchable IDs reconnect physical finds to their digital record

The main design constraint was field reliability: the app had to keep collecting data even when the network disappeared, which is why I treated offline behavior as the core architecture rather than a bonus feature.

Hardest Challenge

The guided voice form — the app asks questions, the user answers by speaking. Getting Web Speech API to work consistently across iOS and Android was brutal. Safari handles audio permissions completely differently from Chrome, and getting continuous recognition to not cut off mid-answer took weeks of refinement. We got it working.

What I Learned

Building something real forces you to understand things you'd never look up voluntarily. I had zero web experience when this started — none of my teammates did either. The constraint of needing it to actually function in a field environment taught me more than any tutorial ever could.

Limitations I'd Fix

Data lives in localStorage — gets lost if cleared. A production version would need a sync queue, remote database, user accounts, and conflict resolution for two people editing the same artifact offline. I would likely use an event-log approach first, then compare it with CRDT-style merging if collaboration became central.

02 3D Tool · WebGL · Interactive Systems 2024–25

Pose
Sandbox

The Problem

Digital artists need fast 3D pose references, but many tools are either expensive desktop software or clunky web ports. I wanted a browser-based tool that opens instantly, needs no account, and lets someone pose a character, add references, and export an image without friction.

What I Built

A browser-based 3D posing tool using Three.js. Users can click body joints, manipulate them with transform controls, add simple props, load 2D reference images pinned to the camera, save poses as JSON, and export clean PNG screenshots from the current canvas view.

How The System Works
UI Layertool buttons, mode switching, save/load actions, reference image controls, and export triggers
Interaction Layermouse input is translated into 3D intent through raycasting, selected joints, and transform controls
Render LayerThree.js manages the scene, camera, lights, body parts, props, and continuous redraw loop

The key engineering problem was not merely displaying a 3D model. It was turning a 2D user action — a click or drag on the screen — into a predictable 3D manipulation, while keeping camera movement, joint selection, and object transforms from fighting each other.

Data Flow
Click Raycast Select joint Transform Render Export / Save

A user click is raycast into the scene to detect the closest selectable joint. Once selected, transform controls update that joint's position or rotation, the scene re-renders from the active camera, and the current pose can either be serialized into JSON or captured as a PNG from the canvas.

Why Three.js

Raw WebGL would have forced me to spend most of the project on shaders, buffers, and low-level rendering. Three.js gave me a scene graph, camera controls, lighting, geometry, and loaders, so I could focus on the harder product problem: making 3D interaction feel understandable to a non-technical user.

Engineering Tradeoffs

The first version uses a manually structured joint hierarchy instead of a full skeletal rig. That made the project buildable as a first 3D web app, but it also limits realism: rotations can feel mechanical, and complex poses need more manual adjustment than they would with inverse kinematics.

What I Learned Technically

Interactive tools are mostly state problems. The app has to know what is selected, which mode is active, whether the user is moving the camera or the model, what objects belong in the scene, and what state should be saved or exported. This taught me that UX and architecture are connected: confusing internal state creates confusing user experience.

What I'd Do Differently

I would replace the hardcoded joint system with a proper skeletal rig and inverse kinematics, add undo/redo through state snapshots, separate rendering logic from UI logic more cleanly, and add performance guardrails for heavier models or more complex scenes.

03 E-Commerce · Brand Design · Real Sales 2026

SVVVD
Y2K

The Context

My cousin had a real fashion brand — Y2K-influenced streetwear — and needed a site that matched the energy of the clothes. This wasn't a practice project. Real products, real customers, real money changing hands.

What I Built

Full e-commerce site with cart system, size selection, live stock tracking (Azul colorway sold out), an order pipeline via email form, and a complete brand identity — typography, color palette, copy, layout. 35 pieces across 3 colorways: Red Flag, Azul, Pinked.

The Constraint

Built in a single day. No framework, no template — HTML, CSS, and vanilla JS from scratch. The constraint forced fast prioritization: what does the user need to see first, what builds trust, what gets the sale.

Product Logic
Cartsize and color choices update the order state before checkout
Stockinventory status is surfaced early so users do not choose unavailable items
Orderemail-based pipeline keeps the site simple while still supporting real purchases

This project is less about deep CS and more about product engineering: reducing friction, showing trust signals, and building the smallest system that could support real customer intent.

What It Taught Me

Design decisions are business decisions. Every layout choice, every line of copy, every color — it all affects whether someone buys or bounces. Engineering and design aren't separate disciplines.

04 SaaS · Backend · Deliberate Practice 2026

Workspace
OS

Why I Built It

Every project I'd made before was purely frontend — no database, no auth, no server. I knew that gap would limit me. So I built something specifically to close it. Not because anyone asked, but because I needed to understand how backends actually work.

What I Built

A multi-tenant SaaS workspace system: email/password authentication, workspace creation/deletion, role-based access (Owner vs Member), invite code system, project and task management with priority levels, and a full audit log. Built in a single day.

System Design
Entitiesusers, workspaces, memberships, projects, tasks, audit logs
Relationshipsusers join many workspaces; workspaces own projects; projects own tasks
Invarianta user must never read or mutate data outside their workspace membership

Supabase handles auth and PostgreSQL storage. Row-Level Security enforces workspace-level isolation in the database, not only in the UI. The key insight was that permissions are part of the data model: if the frontend is bypassed, the database should still reject unauthorized access.

What I Learned

The hardest part wasn't the code — it was thinking in data relationships. Who owns what? How do you prevent a Member from deleting an Owner's project? These aren't coding questions, they're systems design questions. That shift in thinking was the whole point of the exercise.

Tradeoffs & Next Step

Using Supabase let me learn auth, relational data, and RLS quickly, but it also couples the project to platform-specific policies. Next I would add real-time collaboration, email-based invites, pagination for large workspaces, and server-side functions for actions that should not live in client code.
