01 PWA · Mobile · FLL Innovation Project 2025–26

Archaeo
Smart

The Problem

Archaeological field teams record findings on paper — fragile, unstructured, impossible to search or share. Our FLL innovation project was to build a solution that actually works in the field, offline, on a phone.

What I Built

A mobile-first Progressive Web App with voice-guided artifact recording, GPS geo-tagging, QR label generation and scanning, canvas drawing for site sketches, CSV data export, map view, and a statistics dashboard — entirely client-side, no server, works offline.

Hardest Challenge

The guided voice form — the app asks questions, the user answers by speaking. Getting Web Speech API to work consistently across iOS and Android was brutal. Safari handles audio permissions completely differently from Chrome, and getting continuous recognition to not cut off mid-answer took weeks of refinement. We got it working.

What I Learned

Building something real forces you to understand things you'd never look up voluntarily. I had zero web experience when this started — none of my teammates did either. The constraint of needing it to actually function in a field environment taught me more than any tutorial ever could.

Limitations I'd Fix

Data lives in localStorage — gets lost if cleared. A proper sync layer with conflict resolution (CRDTs or similar) would make this production-ready. The voice form also still breaks on some Android builds.

02 3D Tool · WebGL · First Website Ever 2024–25

Pose
Sandbox

The Problem

Digital artists need 3D pose references but existing tools are expensive desktop software or clunky web ports. I wanted something fast, clean, and available the moment you open a browser tab — no install, no account.

What I Built

A browser-based 3D character posing tool with Three.js. Users click joints, use a gizmo to rotate/move/scale, add props (cubes/spheres), load 2D reference images pinned to the camera, save poses as JSON, and export clean PNG screenshots instantly.

Why Three.js

Raw WebGL was too low-level for a first 3D project. Three.js gave me a scene graph, lighting, and camera controls without writing shaders — which let me focus on the actual UX problem: making joint selection and manipulation feel intuitive in 3D space.

Honest Outcome

Nobody bought the Gumroad pose pack. But designing a product, writing landing page copy, setting up pricing, and thinking about user acquisition taught me more about how software products work than the code itself did.

What I'd Do Differently

The joint hierarchy is hardcoded — a proper skeletal rig with inverse kinematics would feel far more natural. I also underestimated how much users would need undo/redo history.

04 SaaS · Backend · Deliberate Practice 2026

Workspace
OS

Why I Built It

Every project I'd made before was purely frontend — no database, no auth, no server. I knew that gap would limit me. So I built something specifically to close it. Not because anyone asked, but because I needed to understand how backends actually work.

What I Built

A multi-tenant SaaS workspace system: email/password authentication, workspace creation/deletion, role-based access (Owner vs Member), invite code system, project and task management with priority levels, and a full audit log. Built in a single day.

Architecture

Supabase handles auth and a PostgreSQL database. Row-Level Security policies enforce that users can only access data in workspaces they belong to — security enforced at the database level, not just the frontend. This was the key insight: data access isn't just a UI problem.

What I Learned

The hardest part wasn't the code — it was thinking in data relationships. Who owns what? How do you prevent a Member from deleting an Owner's project? These aren't coding questions, they're systems design questions. That shift in thinking was the whole point of the exercise.

What I'd Add

Real-time collaboration using Supabase's websocket subscriptions, so multiple users see updates live without refreshing. Proper email-based invites rather than manual code sharing.
